# 04. Проектирование базы данных


## 1. Основные таблицы

- `users`
- `halls`
- `seats`
- `movies`
- `sessions`
- `bookings`
- `tickets`


## 2. Таблица `users`

### Назначение
Хранение учетных записей администраторов системы.

### Основные поля
- `id`
- `email`
- `password_hash`
- `role`
- `is_active`
- `created_at`
- `updated_at`

### Основные ограничения
- `id` — первичный ключ;
- `email` — уникальное значение;
- `role` — ограниченный набор допустимых значений;
- `password_hash` — обязательное поле.

### Комментарий
Для первой версии проекта наличие отдельной регистрации администратора не является обязательным. Администраторские записи могут создаваться вручную.


## 3. Таблица `halls`

### Назначение
Хранение информации о кинозалах.

### Основные поля
- `id`
- `name`
- `rows_count`
- `seats_per_row`
- `is_active`
- `created_at`
- `updated_at`

### Основные ограничения
- `id` — первичный ключ;
- `name` — уникальное значение;
- `rows_count` > 0;
- `seats_per_row` > 0;
- `is_active` — булево значение.

### Комментарий
Поле `is_active` определяет, доступен ли зал для продажи билетов.

## 4. Таблица `seats`

### Назначение
Хранение мест внутри конкретного зала.

### Основные поля
- `id`
- `hall_id`
- `row_number`
- `seat_number`
- `seat_type`

### Основные ограничения
- `id` — первичный ключ;
- `hall_id` — внешний ключ на `halls(id)`;
- `row_number` > 0;
- `seat_number` > 0;
- `seat_type` — ограниченный набор допустимых значений;
- уникальность комбинации (`hall_id`, `row_number`, `seat_number`).

### Комментарий
Уникальность `hall_id + row_number + seat_number`, чтобы в одном зале не существовало 2х одинаковых мест.


## 5. Таблица `movies`

### Назначение
Хранение информации о фильмах.

### Основные поля
- `id`
- `title`
- `description`
- `duration_min`
- `age_rating`
- `poster_url`
- `country`
- `release_date`
- `is_active`
- `created_at`
- `updated_at`

### Основные ограничения
- `id` — первичный ключ;
- `title` — обязательное поле;
- `duration_min` > 0;
- `is_active` — булево значение.

### Комментарий
Один фильм может использоваться в нескольких сеансах.


## 6. Таблица `sessions`

### Назначение
Хранение расписания показов фильмов.

### Основные поля
- `id`
- `movie_id`
- `hall_id`
- `start_at`
- `end_at`
- `base_price`
- `vip_price`
- `status`
- `created_at`
- `updated_at`

### Основные ограничения
- `id` — первичный ключ;
- `movie_id` — внешний ключ на `movies(id)`;
- `hall_id` — внешний ключ на `halls(id)`;
- `start_at` — обязательное поле;
- `end_at` — обязательное поле;
- `end_at` > `start_at`;
- `base_price` >= 0;
- `vip_price` >= 0;
- `status` — ограниченный набор допустимых значений.

### Комментарий
Бизнес-логика, чтобы в одном зале не было пересечений сеансов.


## 7. Таблица `bookings`

### Назначение
Хранение бронирований мест на сеансы.

### Основные поля
- `id`
- `session_id`
- `seat_id`
- `user_id`
- `guest_name`
- `guest_email`
- `guest_phone`
- `booking_code`
- `status`
- `reserved_at`
- `expires_at`
- `created_at`
- `updated_at`

### Основные ограничения
- `id` — первичный ключ;
- `session_id` — внешний ключ на `sessions(id)`;
- `seat_id` — внешний ключ на `seats(id)`;
- `user_id` — внешний ключ на `users(id)`, nullable;
- `booking_code` — уникальное значение;
- `status` — ограниченный набор допустимых значений;
- уникальность комбинации (`session_id`, `seat_id`).

### Комментарий
Уникальность пары (`session_id`, `seat_id`) чтобы не было возможности забронировать одно и то же место на один и тот же сеанс более одного раза (`UNIQUE` и `FOREIGN KEY`).


## 8. Таблица `tickets`

### Назначение
Хранение билетов, сформированных на основе бронирований.

### Основные поля
- `id`
- `booking_id`
- `qr_payload`
- `qr_image`
- `issued_at`

### Основные ограничения
- `id` — первичный ключ;
- `booking_id` — внешний ключ на `bookings(id)`;
- `booking_id` — уникальное значение;
- `qr_payload` — обязательное поле.

### Комментарий
Уникальность `booking_id` обеспечивает связь один-к-одному между бронированием и билетом.


## 9. Связи между таблицами

### 9.1. `halls` — `seats`
- одна запись в `halls` может быть связана со многими записями в `seats`;
- одна запись в `seats` принадлежит одной записи в `halls`.

Тип связи: `1:N`.

Техническая реализация:
- `seats.hall_id` → `halls.id`.

### 9.2. `movies` — `sessions`
- одна запись в `movies` может быть связана со многими записями в `sessions`;
- одна запись в `sessions` относится к одной записи в `movies`.

Тип связи: `1:N`.

Техническая реализация:
- `sessions.movie_id` → `movies.id`.

### 9.3. `halls` — `sessions`
- одна запись в `halls` может быть связана со многими записями в `sessions`;
- одна запись в `sessions` относится к одной записи в `halls`.

Тип связи: `1:N`.

Техническая реализация:
- `sessions.hall_id` → `halls.id`.

### 9.4. `sessions` — `bookings`
- одна запись в `sessions` может быть связана со многими записями в `bookings`;
- одна запись в `bookings` относится к одной записи в `sessions`.

Тип связи: `1:N`.

Техническая реализация:
- `bookings.session_id` → `sessions.id`.

### 9.5. `seats` — `bookings`
- одна запись в `seats` может участвовать во многих записях в `bookings` для разных сеансов;
- одна запись в `bookings` относится к одной записи в `seats`.

Тип связи: `1:N`.

Техническая реализация:
- `bookings.seat_id` → `seats.id`.

### 9.6. `users` — `bookings`
- одна запись в `users` может быть связана со многими записями в `bookings`;
- одна запись в `bookings` может относиться к одной записи в `users` или быть гостевой.

Тип связи: `1:N`.

Техническая реализация:
- `bookings.user_id` → `users.id`;
- поле `bookings.user_id` допускает `NULL`.

### 9.7. `bookings` — `tickets`
- одна запись в `bookings` связана с одной записью в `tickets`;
- одна запись в `tickets` создается для одной записи в `bookings`.

Тип связи: `1:1`.

Техническая реализация:
- `tickets.booking_id` → `bookings.id`;
- `tickets.booking_id` имеет ограничение `UNIQUE`.


## 10. Основные правила целостности

- нельзя создать место без существующего зала;
- нельзя создать сеанс без существующего фильма и зала;
- нельзя создать бронь без существующего сеанса и места;
- нельзя создать билет без существующей брони;
- нельзя создать два одинаковых места в одном зале;
- нельзя создать две брони одного и того же места на один и тот же сеанс;
- нельзя задать отрицательную цену;
- нельзя задать дату окончания сеанса раньше даты начала.
