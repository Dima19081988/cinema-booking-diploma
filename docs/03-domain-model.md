# 03. Предметная модель

## 1. Основные сущности

- User
- Hall
- Seat
- Movie
- Session
- Booking
- Ticket

## 2. Описание сущностей

### 2.1. User

Сущность пользователя системы.

Используется для доступа к административной части и разграничения прав.

#### Основные поля
- `id`
- `email` или `login`
- `password_hash`
- `role`
- `is_active`
- `created_at`
- `updated_at`

#### Назначение
- хранение учетной записи;
- аутентификация;
- определение роли пользователя.

#### Возможные роли
- `ADMIN`
- `GUEST`

Примечание: гость может не иметь отдельной учетной записи в базе данных, если в первой версии проекта бронирование доступно без регистрации. В этом случае роль `GUEST` используется на уровне логики доступа, а не как обязательная запись пользователя.

### 2.2. Hall

Сущность кинозала.

Описывает зал, в котором проводятся киносеансы.

#### Основные поля
- `id`
- `name`
- `rows_count`
- `seats_per_row`
- `is_active`
- `created_at`
- `updated_at`

#### Назначение
- хранение информации о зале;
- определение размеров зала;
- управление доступностью зала для продажи билетов.

#### Бизнес-смысл поля `is_active`
Поле `is_active` показывает, доступен ли зал для продажи билетов.

Если зал неактивен, билеты на связанные с ним сеансы не должны быть доступны гостям.

### 2.3. Seat

Сущность зрительского места.

Описывает конкретное место внутри кинозала.

#### Основные поля
- `id`
- `hall_id`
- `row_number`
- `seat_number`
- `seat_type`

#### Назначение
- хранение схемы зала;
- определение ряда и номера места;
- определение типа места.

#### Возможные типы мест
- `STANDARD`
- `VIP`

Одно место принадлежит одному залу.  
Один зал содержит множество мест.

### 2.4. Movie

Сущность фильма.

Содержит информацию о фильме, которую администратор заполняет в системе.

#### Основные поля
- `id`
- `title`
- `description`
- `duration_min`
- `age_rating`
- `poster_url` или `poster`
- `country`
- `release_date`
- `is_active`
- `created_at`
- `updated_at`

#### Назначение
- хранение информации о фильмах;
- отображение фильмов в афише;
- связывание фильмов с сеансами.

Один фильм может быть связан с несколькими сеансами.

### 2.5. Session

Сущность сеанса.

Описывает конкретный временной промежуток показа фильма в определенном зале.

#### Основные поля
- `id`
- `movie_id`
- `hall_id`
- `start_at`
- `end_at`
- `base_price`
- `vip_price`
- `status`
- `created_at`
- `updated_at`

#### Назначение
- хранение расписания показов;
- определение времени показа фильма;
- определение стоимости мест на сеанс.

#### Возможные статусы
- `DRAFT`
- `PUBLISHED`
- `CANCELED`

Один сеанс относится к одному фильму и одному залу.  
Один фильм может иметь много сеансов.  
Один зал может иметь много сеансов.

### 2.6. Booking

Сущность бронирования.

Описывает факт резервирования конкретного места на конкретный сеанс.

#### Основные поля
- `id`
- `session_id`
- `seat_id`
- `user_id` nullable
- `guest_name`
- `guest_email`
- `guest_phone`
- `booking_code`
- `status`
- `reserved_at`
- `expires_at`
- `created_at`
- `updated_at`

#### Назначение
- хранение информации о брони;
- фиксация выбранного места;
- хранение контактных данных гостя;
- связывание сеанса, места и билета.

#### Возможные статусы
- `RESERVED`
- `CONFIRMED`
- `CANCELED`

Одна бронь относится к одному сеансу и одному месту.

### 2.7. Ticket

Сущность билета.

Описывает итоговый билет, сформированный на основе бронирования.

#### Основные поля
- `id`
- `booking_id`
- `qr_payload`
- `qr_image`
- `issued_at`

#### Назначение
- хранение итогового билета;
- хранение QR-кода;
- хранение данных, подтверждающих бронирование.

Один билет создается на основе одной брони.

## 3. Связи между сущностями

### 3.1. Hall — Seat
- один `Hall` содержит много `Seat`;
- одно `Seat` принадлежит одному `Hall`.

Тип связи: `1:N`.

### 3.2. Movie — Session
- один `Movie` связан с множеством `Session`;
- один `Session` относится к одному `Movie`.

Тип связи: `1:N`.

### 3.3. Hall — Session
- один `Hall` связан с множеством `Session`;
- один `Session` проходит в одном `Hall`.

Тип связи: `1:N`.

### 3.4. Session — Booking
- один `Session` может иметь множество `Booking`;
- одна `Booking` относится к одному `Session`.

Тип связи: `1:N`.

### 3.5. Seat — Booking
- одно `Seat` может участвовать во множестве `Booking` в разные сеансы;
- одна `Booking` относится к одному `Seat`.

Тип связи: `1:N`.

### 3.6. Booking — Ticket
- одна `Booking` связана с одним `Ticket`;
- один `Ticket` создается для одной `Booking`.

Тип связи: `1:1`.

### 3.7. User — Booking
- один `User` может иметь множество `Booking`;
- одна `Booking` может относиться к одному `User` или быть гостевой.

Тип связи: `1:N` с возможностью `NULL` у `user_id`.


## 4. Основные бизнес-правила предметной области

В рамках предметной модели фиксируются следующие правила:
- место существует только внутри конкретного зала;
- сеанс всегда связан с одним фильмом и одним залом;
- бронь всегда относится к одному сеансу и одному месту;
- билет создается только на основе существующей брони;
- зал может быть активным или неактивным;
- продажа билетов доступна только для активных залов и актуальных сеансов;
- одно и то же место не должно быть дважды забронировано на один и тот же сеанс.
