# 02. Архитектура


## 1. Архитектурный стиль

Проект представляет собой fullstack-веб-приложение с клиент-серверной архитектурой.

Система состоит из следующих частей:
- backend API;
- frontend SPA;
- база данных PostgreSQL.

Backend реализует бизнес-логику, валидацию, аутентификацию, авторизацию и работу с данными.

Frontend реализует пользовательский интерфейс, навигацию по страницам и взаимодействие с backend API.


## 2. Общая структура системы

### БД
- пользователи;
- залы;
- места;
- фильмы;
- сеансы;
- бронирования;
- билеты.

### Backend
- принимает HTTP-запросы;
- валидирует данные;
- проверяет права доступа;
- выполняет бизнес-правила;
- работает с базой данных;
- формирует JSON-ответы.

### Frontend
- афиша фильмов;
- расписание сеансов;
- схема кинозала;
- формы бронирования;
- административные экраны.

### Дополнительно
- JWT-аутентификация для административной части
- QR-коды для билетов
- собственные UI-компоненты интерфейса


## 3. Архитектура backend

### backend-модули
- `users`
- `movies`
- `cinema`
- `bookings`
- `common`

### Назначение модулей

#### `bookings`

#### `common`
Содержит:
- общие константы;
- обработку ошибок;
- утилиты;
- общие базовые компоненты backend.

Основные задачи frontend:

- отображение ошибок и статусов;

### Основные frontend-разделы

- управление сеансами;
- просмотр броней.


## 5. Взаимодействие frontend и backend

Обмен данными между frontend и backend выполняется через REST API.

Общая схема взаимодействия:
1. Пользователь выполняет действие в интерфейсе.
2. Frontend формирует HTTP-запрос.
3. Backend принимает запрос.
4. Backend выполняет аутентификацию и проверку прав доступа.
5. Backend валидирует входные данные.
6. Backend выполняет бизнес-логику.
7. Backend сохраняет или получает данные из PostgreSQL.
8. Backend возвращает JSON-ответ.
9. Frontend отображает результат пользователю.


## 6. Аутентификация и авторизация

Для административной части используется аутентификация.

В системе предусмотрены две роли:
- гость;
- администратор.

Гость:
- не требует входа в систему;
- имеет доступ только к публичной части.

Администратор:
- проходит аутентификацию;
- получает доступ к административной части;
- может изменять данные системы.

Регистрация администратора не является обязательной. Администраторские учетные записи могут быть созданы заранее вручную через базу данных, миграции или административную панель.


## 7. Основные бизнес-сценарии

### 7.1. Просмотр фильмов и расписания
1. Гость открывает сайт.
2. Система отображает список фильмов.
3. Гость открывает страницу фильма.
4. Система показывает доступные сеансы.

### 7.2. Бронирование билета
1. Гость выбирает фильм.
2. Гость выбирает сеанс.
3. Система отображает схему зала и свободные места.
4. Гость выбирает место.
5. Frontend отправляет запрос на бронирование.
6. Backend проверяет:
   - существует ли сеанс;
   - принадлежит ли место нужному залу;
   - не занято ли место;
   - можно ли бронировать билет на этот сеанс.
7. Backend создает бронь.
8. Backend формирует билет и QR-код.
9. Frontend отображает подтверждение бронирования.

### 7.3. Управление залом администратором
1. Администратор входит в систему.
2. Администратор создает или редактирует зал.
3. Администратор задает параметры зала:
   - количество рядов;
   - количество мест;
   - типы мест;
   - активность зала.
4. Администратор открывает или приостанавливает продажу билетов для зала.

### 7.4. Управление сеансами
1. Администратор создает фильм.
2. Администратор создает сеанс.
3. Администратор указывает:
   - фильм;
   - зал;
   - дату и время начала;
   - дату и время окончания;
   - стоимость обычных мест;
   - стоимость VIP-мест.
4. Система сохраняет сеанс только при отсутствии конфликта по времени в этом же зале.


## 8. Бизнес-правила системы

В системе должны соблюдаться следующие правила:
- одно место не может быть забронировано более одного раза на один и тот же сеанс;
- билет действителен только для своего сеанса;
- место должно принадлежать залу, в котором проходит выбранный сеанс;
- сеансы в одном зале не должны пересекаться по времени;
- зал по умолчанию может быть неактивным;
- продажа билетов доступна только для активных залов и актуальных сеансов;
- административные действия доступны только администратору;
- все изменяющие операции должны проходить серверную валидацию.


## 9. Модель данных на уровне архитектуры

Основные сущности:
- User
- Hall
- Seat
- Movie
- Session
- Booking
- Ticket

Связи между сущностями:
- один зал содержит множество мест;
- один фильм связан с множеством сеансов;
- один сеанс проходит в одном зале;
- одна бронь относится к одному сеансу и одному месту;
- один билет создается на основе одной брони.


## 10. Требования к интерфейсу

Поскольку оценивается не только функциональность, но и удобство использования, интерфейс должен соответствовать следующим правилам:
- действия должны иметь понятные названия;
- кнопки должны явно отражать результат действия;
- сложные или неочевидные элементы должны сопровождаться подсказками;
- интерфейс должен содержать понятные сообщения об ошибках;
- формы должны иметь валидацию и понятную обратную связь;
- модальные окна должны использоваться для действий, требующих подтверждения или дополнительного ввода.

Модальные окна могут использоваться на основе материалов из предоставленной вёрстки проекта, включая файлы с суффиксом `_popup`.
